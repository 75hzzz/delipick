# 추천 알고리즘 문서

Delipick 백엔드의 사용자 자연어 추천 로직을 발표나 코드 설명에서 사용할 수 있도록 정리한 자료. 현재 추천 시스템은 사용자 자연어 입력, 메뉴 임베딩, 맛 벡터, 리뷰 기반 경험 임베딩, 배달/가격/식당 평점 정보를 함께 사용.

## 구현 파일 구조

추천 API 흐름과 계산식, 키워드 룰을 분리해서 관리.

| 파일 | 역할 |
| --- | --- |
| `main.py` | FastAPI 엔드포인트, DB 조회, 응답 조립 담당 |
| `recommendation_scoring.py` | 추천 점수 계산식, 맛조절 점수, 직접 매칭/상황 보정 계산 담당 |
| `recommendation_rules.py` | 자연어 키워드 확장, 상황 보정 프로필, 맛조절 범위값 관리 |
| `recommend_logic.py` | 자연어 임베딩, 맛 벡터 생성, 배달 대기시간 계산 담당 |

## 1. 추천 점수 전체 구조

최종 추천 점수는 네 가지 기본 축을 합산한 뒤, 직접 음식명 매칭 보너스와 상황 보정 점수를 추가해 계산.

```text
final_score =
    delivery_score * W_delivery                  # 배달 편의성 점수 * 배달 가중치
  + price_score * W_price                        # 가격 적합도 점수 * 가격 가중치
  + restaurant_review_score * W_review           # 음식점 리뷰 점수 * 음식점 리뷰 가중치
  + preference_score * W_preference              # 사용자 자연어 선호도 점수 * 선호도 가중치
  + exact_match_bonus                            # 음식명이 직접 일치할 때 추가 보너스
  + related_match_bonus                          # 관련 음식군이 매칭될 때 추가 보너스
  + context_adjustment                           # 배탈/해장/아플 때 같은 상황 보정 점수
```

최종 점수는 0.0 ~ 1.0 범위로 보정.

## 2. 사용자 유형별 최종 가중치

사용자가 선택한 추천 유형에 따라 최종 점수에서 중요 항목 변경.

| 추천 유형 | 배달 점수 | 가격 점수 | 식당 평점 | 선호도 점수 |
| --- | ---: | ---: | ---: | ---: |
| 편의형 | 40% | 15% | 15% | 30% |
| 미식형 | 10% | 10% | 15% | 65% |
| 가성비형 | 10% | 45% | 15% | 30% |
| 기본형 | 15% | 15% | 15% | 55% |

## 3. 선호도 점수 구성

선호도 점수는 사용자 자연어와 메뉴 데이터를 비교해 계산.

```text
preference_score =
    taste_similarity * taste_weight              # 맛 벡터 유사도 * 맛 가중치
  + menu_embedding_similarity * menu_weight      # 메뉴 임베딩 유사도 * 메뉴 가중치
  + review_embedding_similarity * review_weight  # 리뷰 경험 임베딩 유사도 * 리뷰 가중치
  + manual_taste_score * manual_taste_weight     # 직접 선택한 맛조절 점수 * 맛조절 가중치
```

세부 의미:

| 항목 | 의미 |
| --- | --- |
| taste_similarity | 사용자 입력에서 추출한 맛 벡터와 메뉴 맛 벡터의 유사도 |
| menu_embedding_similarity | 사용자 자연어 임베딩과 메뉴명/메뉴 설명 임베딩의 유사도 |
| review_embedding_similarity | 사용자 자연어 임베딩과 리뷰 기반 경험 임베딩의 유사도 |
| manual_taste_score | 사용자가 직접 선택한 맛조절 범위와 메뉴 맛 값의 적합도 |

## 4. 맛 벡터 계산

맛 벡터는 5가지 맛 요소 사용.

```text
[짠맛, 단맛, 신맛, 감칠맛, 매운맛]
```

기본 메뉴 맛 벡터는 DB에 저장된 메뉴별 맛 값.

```text
base_taste_vector = [salty, sweet, sour, umami, spicy]
```

리뷰에서 보정된 맛 값이 존재하면 추천 계산에서는 리뷰 보정 맛 벡터를 우선 사용.

```text
menu_taste_vector =
    review_taste_vector가 존재하면 review_taste_vector
    아니면 base_taste_vector
```

## 5. 맛 유사도 공식

맛 유사도는 유클리드 거리 기반 유사도와 코사인 유사도를 함께 사용.

```text
taste_similarity =
    euclidean_similarity * 0.65
  + cosine_similarity * 0.35
```

유클리드 유사도는 두 맛 벡터의 거리 차이 판단. 코사인 유사도는 두 벡터의 방향 유사성 판단.

두 방식을 섞는 이유는 단순히 "매운 정도가 비슷한가"뿐 아니라, 전체적인 맛 성향의 방향성까지 함께 반영하기 위함.

## 6. 맛조절 선택 보정

사용자가 맛조절을 건드리지 않으면 기본값은 `선택 안 함`. 이 경우 해당 맛은 추천 계산에 직접 개입하지 않음.

맛조절 단계:

| 단계 | 내부 값 | 적용 범위 |
| --- | ---: | --- |
| 선택 안 함 | -1 | 계산 제외 |
| 낮음 | 0 | 0.0 ~ 0.4 |
| 중간 | 1 | 0.3 ~ 0.7 |
| 높음 | 2 | 0.6 ~ 1.0 |

선택된 맛은 두 방식으로 반영.

```text
1. 사용자 맛 벡터 목표값 보정
   낮음 = 0.2
   중간 = 0.5
   높음 = 0.8

2. 메뉴 맛 값의 범위 적합도 점수 추가
   범위 안 = 1.0
   범위 밖 = 거리 기반 부드러운 감점
```

범위 밖 감점 공식:

```text
manual_taste_score =
    1 - (범위에서 벗어난 거리 / 0.35)
```

점수는 0.0 ~ 1.0 범위로 보정. 완전 제외 필터가 아니라 점수 보정 방식이라 맛 데이터가 조금 부정확해도 추천 후보가 바로 탈락하지 않음.

맛조절 가중치:

```text
manual_taste_weight = min(0.30, 0.16 + 선택한_맛_개수 * 0.04)
```

예시:

```text
사용자 입력: "마라탕"
맛조절: 매운맛 낮음

=> 마라탕 직접 매칭 유지
=> 너무 매운 메뉴 감점
=> 매운 정도가 낮거나 중간에 가까운 마라 메뉴 상대적 상승
```

```text
사용자 입력 없음
맛조절: 단맛 높음

=> 자연어 임베딩 없이 단맛 높은 메뉴 중심으로 선호도 점수 계산
```

## 7. 리뷰 임베딩의 의미

현재 DB의 `review_embedding`은 단순히 리뷰 원문 전체를 붙여서 임베딩한 값이 아님.

현재 의미:

```text
review_embedding = 리뷰에서 추출한 상황 / 감정 / 사용자 경험 중심 임베딩
```

리뷰에서 아래 요소를 중심으로 임베딩.

| 요소 | 예시 |
| --- | --- |
| 상황 | 혼밥, 야식, 해장, 가족 식사, 아플 때 |
| 감정 | 든든함, 만족감, 부담 없음, 자극적임 |
| 경험 | 배달 상태, 양, 먹기 편함, 재주문 의사 |

맛 자체는 메뉴 맛 벡터가 담당. 리뷰 임베딩은 "이 메뉴가 어떤 상황에 어울리는지"를 보정하는 역할.

## 8. 상황형 입력 판단

사용자 입력이 단순 음식명인지, 상황/컨디션 기반 요청인지 판단하기 위해 경험 의도 점수 `E` 계산.

```text
E = experience_intent_score
```

`E`는 0.0 ~ 1.0 범위.

| E 값 | 의미 |
| --- | --- |
| 0에 가까움 | "마라탕", "구운치킨"처럼 직접 음식명 중심 |
| 1에 가까움 | "배탈났어", "해장하고 싶어", "아이랑 먹기 좋은 메뉴"처럼 상황 중심 |

주요 상황 키워드:

```text
스트레스, 기분, 혼밥, 야식, 배부, 든든,
아이, 가족, 아플, 감기, 속, 배탈, 해장,
술, 입맛, 선물, 간식, 달달, 부담
```

기본 계산식:

```text
raw_E = 0.35 + 상황키워드_개수 * 0.20
E = raw_E * (1 - direct_intent * 0.55)
```

직접 음식명 의도가 강하면 `direct_intent` 상승. 이 경우 상황형 점수 `E`는 감소. 예를 들어 "마라탕"은 상황형 입력으로 과하게 해석하지 않고 음식명 직접 검색에 가깝게 처리.

## 9. 선호도 내부 동적 가중치

선호도 점수 내부에서는 상황형 입력 여부에 따라 맛/메뉴/리뷰 임베딩의 비중 변경.

```text
taste_weight  = 0.45 - 0.05 * E
menu_weight   = 0.35 - 0.05 * E
review_weight = (0.20 + 0.10 * E) * (1 - direct_intent * 0.45)
```

정리:

| 입력 유형 | 맛 벡터 | 메뉴 임베딩 | 리뷰 경험 임베딩 |
| --- | ---: | ---: | ---: |
| 음식명 직접 입력 | 약 45% | 약 35% | 약 20% 이하 |
| 상황형 입력 | 약 40% | 약 30% | 최대 약 30% |

"마라탕"처럼 음식명이 명확하면 메뉴명/맛 쪽 비중 상승. "배탈났어", "해장하고 싶어"처럼 상황이 중요하면 리뷰 경험 임베딩 영향 상승.

## 10. 직접 음식명 매칭 보너스

임베딩만 사용하면 "구운치킨" 검색에서 베이글이나 치즈볼처럼 의미상 일부만 비슷한 메뉴가 상위 노출될 수 있음. 이를 줄이기 위해 직접 텍스트 매칭 보너스 추가.

```text
exact_match_bonus = exact_match * direct_intent * 0.22
related_match_bonus = related_match * (direct_intent * 0.06 + (1 - direct_intent) * 0.10)
```

| 보너스 | 의미 |
| --- | --- |
| exact_match_bonus | 사용자 입력 음식명이 메뉴명에 직접 포함될 때 강하게 가산 |
| related_match_bonus | 비슷한 음식군이나 확장 키워드가 매칭될 때 약하게 가산 |

예: "마라탕" 검색 시 메뉴명에 "마라탕"이 직접 포함된 메뉴가 우선 노출되도록 보정.

## 11. 컨디션/상황 보정

특정 상황에서는 임베딩 유사도만으로 부적절한 메뉴가 추천될 수 있음. 예를 들어 "나 어제 배탈났어"라는 입력에 아메리카노가 추천되는 것은 사용자 의도와 맞지 않음.

이를 방어하기 위해 상황별 보정 점수 추가.

```text
context_adjustment = 상황별 boost 점수 - 상황별 penalty 점수
```

보정 범위:

```text
-0.42 <= context_adjustment <= +0.24
```

대표 보정 프로필 예시:

```text
실제 구현에서는 유사한 생활 표현을 같은 의도군으로 묶어 처리.
예: "뜨끈한 밥", "든든한 한 끼", "밥류"는 따뜻한 식사/밥류 의도군으로 처리.
```

| 상황 | 상승 메뉴 예시 | 하락 메뉴 예시 |
| --- | --- | --- |
| 배탈/소화불량 | 죽, 계란찜, 우동, 쌀국수, 맑은 국물, 담백한 메뉴 | 커피, 카페인, 마라, 매운 음식, 튀김, 치킨, 버거 |
| 아플 때 | 죽, 설렁탕, 갈비탕, 국, 탕, 수프, 보양식 | 커피, 차가운 음료, 마라, 매운 음식, 튀김 |
| 해장 | 해장국, 국밥, 감자탕, 설렁탕, 찌개, 짬뽕, 라면 | 디저트, 케이크, 도넛, 커피, 과한 마라 |
| 따뜻한 식사/밥류 | 밥, 국밥, 덮밥, 백반, 찌개, 탕, 설렁탕, 비빔밥, 우동 | 초밥, 회, 도넛, 베이글, 디저트, 아이스, 샐러드 |
| 차가운 음료 | 아이스커피, 라떼, 에이드, 스무디, 주스, 아이스티, 밀크티 | 닭발, 주먹밥, 세트, 식사+음료 조합, 밥/국수/탕류 |
| 아이/가족 | 순한 메뉴, 돈까스, 죽, 밥, 베이글, 설렁탕 | 카페인, 마라, 매운 음식, 술 관련 메뉴 |
| 스트레스/자극 | 매운 메뉴, 마라, 떡볶이, 닭발, 치킨, 청양, 짬뽕 | 죽, 커피처럼 의도와 약한 메뉴 |

## 12. 가격 점수

가격 점수는 사용자 유형에 따라 의미 변경.

가성비형에서는 낮은 가격대 메뉴의 점수 상승. 반대로 미식형에서는 가격보다 취향과 메뉴 적합도가 더 중요하므로 가격 가중치 낮게 설정.

```text
budget 유형:
price_score 영향 45%

gourmet 유형:
price_score 영향 10%
```

## 13. 배달 점수

배달 점수는 편의형에서 가장 크게 반영.

```text
convenience 유형:
delivery_score 영향 40%
```

사용자가 "빨리", "가볍게", "지금 바로"처럼 편의성을 기대하는 유형에서는 배달 시간이나 배달 관련 점수가 추천 순위에 더 큰 영향.

## 14. 장바구니 예상 소요 시간

장바구니에서는 주문 수량 증가에 따른 간단한 조리 지연을 추가 반영.

```text
cart_estimated_time =
    base_delivery_time
  + additional_prep_delay
```

추가 조리 지연:

```text
additional_prep_delay =
    min((total_count - 1) * 2, 10)
```

계산 예시:

| 총 수량 | 추가 지연 |
| ---: | ---: |
| 1개 | +0분 |
| 2개 | +2분 |
| 3개 | +4분 |
| 4개 | +6분 |
| 5개 | +8분 |
| 6개 이상 | +10분 |

주문량이 늘어나면 조리 준비 시간이 늘어난다고 가정. 과도한 시간 증가 방지를 위해 최대 10분으로 제한.

## 15. 예시 결과 흐름

### "마라탕"

```text
직접 음식명 의도 높음
상황형 점수 낮음
exact_match_bonus 강하게 적용
```

결과적으로 메뉴명에 "마라탕"이 직접 들어간 메뉴가 우선 노출.

### "나 어제 배탈났어"

```text
상황형 점수 높음
리뷰 경험 임베딩 비중 증가
배탈/소화불량 context_adjustment 적용
커피, 마라, 튀김류 penalty
죽, 맑은 국물, 담백한 메뉴 boost
```

결과적으로 아메리카노처럼 컨디션에 맞지 않는 메뉴는 하락. 죽/국물/담백한 메뉴는 상승.

### "해장하기 좋은 국물 메뉴"

```text
상황형 점수 높음
해장 프로필 적용
국밥, 해장국, 찌개, 짬뽕, 라면 계열 boost
디저트/커피류 penalty
```

### "뜨근한 밥먹고 싶네"

## 16. 현재 방식의 장점

현재 방식은 단순 임베딩 검색 대비 아래 부분 개선.

1. 음식명이 직접 들어간 검색에서 정확도 상승
2. "배탈", "아플 때", "해장" 같은 컨디션 입력 방어
3. 리뷰 임베딩을 맛 판단이 아니라 상황/감정/경험 판단에 사용
4. 사용자 유형별로 배달, 가격, 취향 비중을 다르게 적용
5. 사용자가 직접 선택한 맛조절만 추가 반영
6. 임베딩 점수와 규칙 기반 보정을 함께 사용해 예외 상황 감소

## 17. 한계

1. 맛 벡터와 리뷰 보정값은 데이터 품질에 따라 추천 정확도 영향
2. 자연어 입력에서 추출한 맛 벡터가 잘못 해석되면 일부 추천 순위 변동 가능
3. 음식명, 조리 방식, 상황 표현이 매우 복합적인 경우 규칙 기반 보정만으로 완전한 해석 어려움
4. 실제 주문/클릭 이력 기반 개인화 학습은 현재 범위에 포함되지 않음
